# OSV 本地漏洞扫描管理器 - 需求背景与定位

## 1. 背景概述
随着开源软件的广泛应用，第三方依赖项引入的安全漏洞（Vulnerabilities）已成为软件供应链安全的最大威胁之一。Google 开源的 **OSV.dev** 漏洞数据库及 **osv-scanner** 命令行工具，通过对包名、版本及具体 Commit Hash 进行精准匹配，为开发者提供了快速且免费的开源漏洞扫描方案。

然而，`osv-scanner` 原生是以纯 CLI（命令行界面）形式提供服务的。虽然非常适合 CI/CD 自动化流水线，但在**本地个人开发机/工作站环境**下，纯命令行工具的使用体验存在较大割裂，难以帮助开发者进行常态化、多项目的供应链安全治理。

为了将安全漏洞排查真正融入日常开发流程中，我们计划在本地工具集 `01_rust_tool` 中，新增一个 **OSV 本地漏洞扫描管理器 (OSV Dependency Security Manager)** 模块，解决本地依赖安全治理的痛点。

---

## 2. 核心痛点分析
当前在本地计算机上直接使用 `osv-scanner` CLI 存在以下显著痛点：

### 2.1 多项目管理碎片化，维护成本高
* **现状**：开发者计算机中通常存放着几十甚至上百个大小项目，分布在不同的目录中（例如前端 Node.js 项目、后端 Rust/Go 项目等）。
* **痛点**：要对所有项目进行漏洞体检，必须手动切换到各个目录执行 `osv-scanner scan source -r .`。即使写脚本进行全盘递归，也会因为扫描了无用的 `node_modules`、`target` 目录或临时归档文件而导致扫描极慢，且难以分类管理。

### 2.2 漏洞库动态更新，扫描机制过于被动
* **现状**：新的漏洞（CVE / GHSA）每天都在被不断披露和收录。
* **痛点**：今日编译安全的依赖项，在数天或数周后可能就会产生新的已知漏洞。纯 CLI 扫描是被动的、单次触发的，开发者不可能每天手动去轮询扫描本地的所有项目。缺乏**后台静默守护**与**主动预警（系统通知）**机制，使得本地历史项目极易沦为安全死角。

### 2.3 修复决策摩擦大，操作门槛高
* **现状**：OSV 终端输出包含大量 CVE/GHSA 编号及繁杂的依赖树路径，对于不熟悉安全背景的开发者来说，排查困难。
* **痛点**：
  * **修复难**：开发者难以快速识别哪些漏洞可以通过升级直接无损修复，哪些需要手动修改源码。
  * **忽略难**：对于确认无安全风险的漏洞（例如仅在开发环境中使用的测试包漏洞），在 CLI 下需要手动繁琐地编写或修改各个项目根目录下的 `osv-scanner.toml` 过滤配置文件。

### 2.4 缺乏全局可视化的健康看板
* **现状**：命令行输出是一次性的控制台文本，难以长久保存或直观比对。
* **痛点**：无法直观给出计算机中所有项目的“整体安全态势评分”，不便于开发者快速识别最紧急的 Critical 漏洞并进行优先级排序。

---

## 3. 建设目标 (Goals)
本模块的设计目标是建立一个**轻量、无感、高效**的本地开源依赖安全防护网：

* **统一监控与订阅**：提供图形界面或集中式配置，支持对本地指定的源码目录进行“一键订阅”，集中监控。
* **后台静默守护**：支持配置轻量级的后台定时扫描任务，发现新漏洞时主动向 macOS 发送系统级别通知。
* **智能漏洞分析与一键修复**：在 UI 侧将扫描结果格式化展示，清晰呈现漏洞危害、受影响路径，并为可直接升级的依赖提供“一键一键应用修复”动作。
* **规则集中化/免密忽略**：提供可视化界面配置特定漏洞的忽略策略，自动在项目本地维护 `osv-scanner.toml`，降低配置摩擦力。
* **极佳的视觉与动效体验**：继承 `01_rust_tool` 的现代化设计语言，采用直观的健康度圆环和发光状态卡片，提供高品质的交互细节。

---

## 4. 目标使用场景 (Scenarios)
* **场景一：新项目一键加入监控**
  开发者新建或克隆了一个项目，将其拖入本工具中，即刻完成首次漏洞扫描并加入每日后台体检清单。
* **场景二：晨间漏洞主动预警**
  后台定时器在凌晨完成扫描，早晨开发者开机时，macOS 右上角弹出通知：“发现 1 个 Critical 严重漏洞，影响 X 项目，点击立即修复”。
* **场景三：快速依赖版本升级**
  开发者在侧拉面板中看清受影响的依赖关系，点击 `Apply Fix`，工具自动在后台运行包管理器更新锁文件，免去手动查找修复版本的繁琐过程。
