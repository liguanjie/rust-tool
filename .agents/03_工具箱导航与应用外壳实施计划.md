# 工具箱导航与应用外壳实施计划

## 模块目标

为 RustTool 建立稳定的应用外壳，让后续工具可以按功能模块接入，而不是把所有页面平铺在一个入口里。

当前产品结构分为两类：

- 工作台：偏本机服务管理，例如 Windows 工作台。
- 工具箱：偏输入、转换、校验、生成类工具，例如 VLESS 转 Mihomo。

## 当前已落地

关键文件：

```text
frontend/src/stores/tools.ts
frontend/src/components/AppSidebar.vue
frontend/src/components/ToolShell.vue
frontend/src/router/index.ts
frontend/src/styles/base.css
frontend/src/pages/WindowsWorkbench.vue
frontend/src/pages/VlessToMihomo.vue
```

已完成能力：

- 左侧导航按模块分组。
- 工作台作为第一个功能入口。
- VLESS 转 Mihomo 归入工具箱，便于后续接入其他工具。
- 页面统一使用 ToolShell 承载标题、描述、内容区。
- 统一按钮、输入框、卡片、抽屉、提示样式。
- Web 预览与 Tauri 桌面版共用同一套前端页面。

## 实施步骤

### 阶段 1：确定信息架构

1. 将工具分为工作台与工具箱。
2. 将本机服务管理放入工作台。
3. 将格式转换、配置生成等能力放入工具箱。
4. 规划工具元数据模型：id、名称、路径、描述、图标。

### 阶段 2：建立导航数据源

1. 在 `frontend/src/stores/tools.ts` 中定义工具分组。
2. 每个工具只通过元数据注册到导航。
3. 避免在侧边栏组件中写死工具列表。
4. 新增工具时优先修改工具注册表，再新增路由与页面。

### 阶段 3：建立统一页面外壳

1. 创建或完善 `ToolShell.vue`。
2. 页面标题、描述、eyebrow 统一由外壳承载。
3. 页面内部只关注业务表单和结果展示。
4. 保持工作台、转换工具、未来工具的视觉一致性。

### 阶段 4：建立路由规范

1. 工作台路由使用 `/tools/...`。
2. 工具箱路由使用 `/toolbox/...`。
3. 首页默认跳转到 Windows 工作台。
4. 每个工具页面独立加载自己的 store。

## 验收标准

- 打开应用后默认进入 Windows 工作台。
- 侧边栏可以清晰看到“工作台”和“工具箱”。
- 点击 VLESS 转 Mihomo 可以进入工具页面。
- 后续新增工具时，不需要改动既有工具核心逻辑。
- Web 开发环境与 Tauri 桌面环境页面一致。

## 后续扩展
