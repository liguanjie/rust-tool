# AI 安全文档模块优化实施计划

## 目标

基于参考图中的“文档库 + Markdown 工作区 + AI / 安全审计侧栏”工作台形态，优化 RustTool 的 AI 安全文档模块，使其从“安全文档编辑与 AI 问答”升级为“安全文档审计闭环”：

1. 文档风险可见：文档列表、正文行号、右侧审计卡片都能显示风险等级和位置。
2. 问题处理可执行：每个风险都有解释、建议、查看、处理、忽略和复核状态。
3. AI 辅助不越权：默认只处理脱敏后的上下文，涉及 secret 明文必须继续受现有保险箱权限约束。
4. Web 与桌面一致：新增接口必须同时覆盖 HTTP、Tauri command 和 `memoApi` 适配层。

## 当前代码锚点

- 页面入口：`frontend/src/pages/AiMemo.vue`
- 前端统一适配：`frontend/src/services/memoApi.ts`
- HTTP 路由：`crates/rust_tool_server/src/routes/memo.rs`
- Tauri command：`frontend/src-tauri/src/memo_commands.rs`
- Tauri 注册：`frontend/src-tauri/src/lib.rs`
- 核心能力：`crates/rust_tool_core/src/memo/mod.rs`
- 本地脱敏：`crates/rust_tool_core/src/memo/redactor.rs`
- Markdown 存储：`crates/rust_tool_core/src/memo/markdown_store.rs`

本计划必须遵守 `.agents/06_桌面与Web一致性规范.md`：页面不得直接散落 `fetch('/api/memo/...')`，所有 AI 安全文档请求统一经 `memoApi`。

## 产品方案

### 1. 版面重组

将当前页面从“聊天 / 编辑 / 预览三列并列”调整为更贴近安全审计工作台的三栏结构：

1. 左侧文档库
   - 保留树形文档列表、搜索、新建、删除。
   - 新增风险等级标签：高危、警告、安全。
   - 新增风险数量摘要，例如 `3`、`高危 1`。
   - 点击文档后自动加载该文档的最近一次审计结果。

2. 中间文档工作区
   - 顶部显示文档标题、文件名、保存、安全分享或导出入口。
   - Markdown 编辑器显示行号。
   - 对风险所在行显示浅色高亮和左侧风险标记。
   - 支持编辑 / 预览切换，避免当前三列在窄屏下过挤。
   - 保留密码箱抽屉，但默认收起，减少审计时的视觉干扰。

3. 右侧智能侧栏
   - 使用标签页：`AI 助手`、`安全审计`、`预览`。
   - `AI 助手` 显示文档摘要、建议任务、快捷操作、对话输入。
   - `安全审计` 显示风险卡片、筛选、状态和批量处理入口。
   - `预览` 显示渲染后的 Markdown，供审阅时快速切换。

### 2. 风险卡片

每个风险卡片最少包含：

- 位置：例如 `L14` 或 `L14-L16`
- 等级：`critical`、`warning`、`info`
- 标题：例如 `硬编码密钥泄露`
- 说明：识别到的风险原因
- 建议：安全处理建议
- 操作：`查看`、`解释`、`处理`、`忽略`
- 状态：`open`、`fixed`、`ignored`、`reviewing`

高危卡片使用红色边框与弱底色，警告使用黄色，信息提示使用中性色。批量处理按钮必须进入确认或 diff 预览，不允许静默改写整篇文档。

### 3. 选区浮动 AI 操作

在 Markdown 编辑区选中文本后显示浮动工具条：

- `AI 改写`：优化表达但保持事实。
- `总结`：生成短摘要。
- `脱敏`：将疑似敏感值替换为 `{{secret:key}}` 占位符，并引导写入 KDBX。
- `解释`：解释选区内容或安全含义。

选区操作只改选区范围。涉及 secret 明文时，继续遵守 `allowAiSecrets = false` 的默认限制。

## 核心数据模型

新增核心类型建议放在 `crates/rust_tool_core/src/memo/audit.rs`：

```rust
pub struct SecurityFinding {
    pub id: String,
    pub doc_id: String,
    pub line_start: usize,
    pub line_end: usize,
    pub severity: FindingSeverity,
    pub kind: FindingKind,
    pub title: String,
    pub detail: String,
    pub evidence: String,
    pub recommendation: String,
    pub status: FindingStatus,
}

pub enum FindingSeverity {
    Critical,
    Warning,
    Info,
}

pub enum FindingStatus {
    Open,
    Fixed,
    Ignored,
    Reviewing,
}
```

前端对应类型建议放在 `AiMemo.vue` 内先就地定义；后续若扩展到多页面，再抽到 `frontend/src/types/memo.ts`。

## 本地审计规则

第一阶段先实现离线规则扫描，不依赖 AI：

1. 硬编码密钥
   - 识别 API Key、token、secret、password、private key、access key。
   - evidence 必须做中间遮罩，例如 `sk-...abcd`，不得在卡片中完整展示。

2. JWT 与签名弱点
   - 识别 `HS256`、`none`、弱密钥描述、JWT 示例中的高风险声明。
   - 给出迁移到 `RS256` / `ES256` 或增强密钥管理的建议。

3. 不安全链接
   - 识别 `http://`、内网明文服务、第三方非 HTTPS 链接。
   - 建议切换到 HTTPS 或内部可信通道。

4. 敏感运维信息
   - 识别数据库连接串、私网 IP + 端口、明文账号、云资源标识。
   - 建议转为占位符或移入密码库。

5. 文档治理问题
   - 识别“临时关闭校验”“跳过鉴权”“TODO: remove auth”等高风险描述。
   - 建议补充审批、有效期、回滚方案。

## 接口设计

新增接口必须同时支持 Web 和桌面。

### HTTP 路由

在 `crates/rust_tool_server/src/routes/memo.rs` 增加：

- `POST /api/memo/audit/scan`
  - 入参：`{ docId: string, markdown?: string }`
  - 出参：`{ findings: SecurityFinding[], summary: AuditSummary }`
  - 用于扫描当前文档；如果传入 markdown，则扫描未保存草稿。

- `POST /api/memo/audit/finding/status`
  - 入参：`{ docId: string, findingId: string, status: string }`
  - 出参：`{ ok: true }`
  - 用于忽略、标记复核、标记已修复。

- `POST /api/memo/audit/fix-preview`
  - 入参：`{ docId: string, findingId: string, markdown: string }`
  - 出参：`{ patchMarkdown: string, explanation: string }`
  - 只生成修复预览，不直接保存。

### Tauri command

在 `frontend/src-tauri/src/memo_commands.rs` 增加对应 command：

- `memo_audit_scan`
- `memo_audit_update_finding_status`
- `memo_audit_fix_preview`

并在 `frontend/src-tauri/src/lib.rs` 的 `tauri::generate_handler!` 注册。

### 前端适配

在 `frontend/src/services/memoApi.ts` 中新增映射：

- `POST /audit/scan` -> `memo_audit_scan`
- `POST /audit/finding/status` -> `memo_audit_update_finding_status`
- `POST /audit/fix-preview` -> `memo_audit_fix_preview`

## 实施阶段

### Phase 1：视觉布局与状态模型

1. 在 `AiMemo.vue` 增加右侧面板状态：
   - `rightPanelTab`
   - `auditFindings`
   - `auditSummary`
   - `selectedFindingId`
   - `auditLoading`
2. 重构 unlocked 后主界面布局为左 / 中 / 右三栏。
3. 中间编辑区增加行号容器和风险行高亮。
4. 右侧增加 `AI 助手`、`安全审计`、`预览` 标签页。
5. 文档列表增加风险等级标签占位，先从前端扫描结果计算。

验收标准：

- 页面解锁后能正常显示文档列表、编辑器、右侧侧栏。
- 小屏下左侧和右侧可折叠，不遮挡编辑器。
- 原有保存、新建、删除、聊天、预览功能不回退。

### Phase 2：本地审计扫描

1. 新增 `crates/rust_tool_core/src/memo/audit.rs`。
2. 实现 Markdown 按行扫描、风险类型识别、evidence 脱敏。
3. 在 `MemoManager` 暴露 `audit_document`。
4. 增加 HTTP 路由、Tauri command、`memoApi` 映射。
5. 在前端打开文档、保存文档后触发扫描。

验收标准：

- 示例文档中硬编码密钥、弱 JWT、HTTP 链接能被识别。
- 风险卡片点击 `查看` 后编辑器滚动到对应行并高亮。
- 扫描不需要调用外部模型。

### Phase 3：AI 辅助任务

1. 将右侧 `AI 助手` 改为文档上下文面板：
   - 文档摘要
   - 建议任务
   - 快捷操作：整理格式、生成摘要、提取待办、一键脱敏、检查风险
2. 复用现有 `runEditorAi` 与 `/chat` 能力，新增安全提示词模板。
3. 实现选区浮动菜单：
   - 追踪 textarea 选区。
   - 对选区执行改写、总结、脱敏、解释。
   - 返回结果前展示确认，用户确认后替换选区。

验收标准：

- 用户可以对选中文本执行 AI 操作。
- AI 操作不会默认读取 KDBX secret 明文。
- 生成的内容能应用到当前 Markdown 并保存。

### Phase 4：修复预览与批量处理

1. 风险卡片增加 `处理` 按钮，生成修复预览。
2. 增加 diff 或左右对比确认弹窗。
3. `处理 3 个风险` 批量按钮只生成候选改动，不直接保存。
4. 处理后自动重新扫描，已消失的 finding 标记为 `fixed`。
5. 支持 `忽略` 并持久化忽略状态。

验收标准：

- 单个风险处理必须经过用户确认。
- 批量处理必须展示改动数量、受影响行和风险类型。
- 取消处理不改变文档内容。

### Phase 5：安全分享与导出

1. 增加“安全分享”入口：
   - 默认导出脱敏 Markdown。
   - 可选导出审计摘要。
   - 不导出 KDBX 明文。
2. 增加审计报告导出：
   - 文档标题、扫描时间、风险摘要、风险列表、处理状态。
   - Markdown 优先，后续再考虑 PDF。
3. 增加状态栏：
   - 保险箱已解锁 / 已锁定。
   - 同步或备份状态。
   - 最近扫描时间。

验收标准：

- 导出内容不包含 secret 明文。
- 报告能复现风险位置、等级和处理状态。

## 样式原则

1. 安全审计 UI 应保持克制、清晰、可扫描，避免营销式大 hero。
2. 使用现有暗色视觉体系，但减少纯装饰背景；重点放在编辑器、卡片、状态标识。
3. 图标优先使用 `@lucide/vue`。
4. 卡片圆角控制在 8px 左右，与现有工具站保持一致。
5. 风险等级颜色：
   - 高危：红色边框 + 淡红背景
   - 警告：琥珀色边框 + 淡黄背景
   - 安全：绿色标签
6. 所有按钮文本必须能在窄宽度容器内完整显示，必要时只显示图标并提供 `title`。

## 测试计划

1. Rust 单元测试
   - `audit.rs` 对每类风险规则添加样例测试。
   - 验证 evidence 遮罩不会泄露完整密钥。
   - 验证行号计算正确。

2. HTTP/Tauri 适配测试
   - Web 路由返回统一 `{ error: { code, message } }`。
   - Tauri command 错误映射与 Web 一致。
   - `memoApi` 覆盖所有新增路径。

3. 前端手动验证
   - 解锁保险箱。
   - 新建或打开包含密钥、HS256、HTTP 链接的 Markdown。
   - 扫描风险、查看定位、解释、处理、忽略。
   - 保存后重新扫描。
   - 导出脱敏报告。

4. 构建验证
   - `pnpm run build`
   - `cargo test`
   - 如改到共享核心或跨端命令，运行 `./rt test`

## 风险与约束

1. 规则扫描会有误报，第一版必须提供 `忽略` 和 `复核` 状态。
2. AI 修复可能改变原始审计表述，必须先预览再应用。
3. 文档中的密钥 evidence 必须遮罩展示，不能在卡片、日志、导出报告中完整出现。
4. 不得把 KDBX 明文 secret 注入 AI 上下文，除非用户显式开启高风险权限。
5. 三栏布局容易拥挤，移动端和窄屏必须降级为可折叠侧栏或标签页。

## 建议开发顺序

1. 先做 Phase 1，让版面结构稳定。
2. 再做 Phase 2，让安全审计能离线闭环。
3. 接入 Phase 3 的 AI 选区和任务能力。
4. 最后做 Phase 4、Phase 5 的修复确认、批量处理和导出。

推荐先交付 Phase 1 + Phase 2 作为 MVP：用户能看到文档风险、定位风险、理解风险并手动处理。AI 修复和报告导出作为第二批功能更稳妥。

## 2026-06-11 实施复盘

07 的计划已进入“可用工作台”状态，不再只是布局设想：

1. 三栏工作台已落地：左侧文档库、中间 Markdown 编辑区、右侧 `AI 助手 / 安全审计 / 预览` 标签页。
2. 文档库已显示风险等级与数量摘要，能区分高危、警告、关注和安全文档。
3. 本地审计扫描已覆盖硬编码密钥、弱 JWT、明文 HTTP、敏感运维信息和高风险治理描述，evidence 会遮罩。
4. 风险卡片已具备查看、解释、处理、复核、忽略以及批量处理入口；批量处理进入确认，不会静默改写。
5. AI 辅助任务已覆盖整理格式、生成摘要、提取待办、一键脱敏、检查风险和选区浮动操作。
6. 密码箱抽屉默认收起；一键脱敏会把真实值导入当前草稿密码箱，但不自动保存。
7. 安全分享导出已默认脱敏，并记录 `safeShareExported` 审计事件。
8. Web HTTP、Tauri command 和 `memoApi` 已保持同构适配。

本阶段结论：07 已达到“AI 安全文档审计工作台”的 MVP 到增强版之间的状态。它已经能支撑团队在单篇或少量文档中发现、理解、修复和脱敏风险，但它本身还不是完整的长期治理系统。

仍需继续打磨的点：

1. 选区浮动工具条的位置和体验还可以更像原生编辑器能力。
2. 风险修复预览还不是完整 diff 视图，目前以候选改动和确认文案为主。
3. 审计规则仍是第一批本地规则，需要根据真实团队文档继续补规则、降误报。
4. 文档层的元数据还偏轻，后续应补 `owners`、`document_type`、`system_tags`、`review_status` 等结构化字段。
