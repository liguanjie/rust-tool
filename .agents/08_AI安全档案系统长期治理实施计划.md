# AI 安全档案系统长期治理实施计划

## 目标定位

RustTool 的 AI 安全文档能力最终目标不是“更好看的 Markdown 编辑器”，而是一个 **持续陪团队做安全治理的 AI 安全档案系统**。

它应当长期沉淀团队的安全知识、风险证据、审计结论、修复过程、复核记录和合规映射，让每一次文档编写、架构调整、接口变更、应急复盘都能进入同一套本地优先的安全档案。

与 `.agents/07_AI安全文档模块优化实施计划.md` 的关系：

- `07` 是工作台 MVP：解决文档编辑、风险扫描、风险卡片、AI 辅助、修复预览和导出。
- `08` 是治理系统路线图：解决团队档案、风险生命周期、知识图谱、版本追踪、规范映射和协作复核。

## 核心产品信念

1. 文档不是静态内容，而是安全资产。
2. 风险不是一次扫描结果，而是有证据、状态、负责人、历史和复核结论的治理对象。
3. AI 不是自动审批者，而是持续帮助整理、解释、关联、比对和生成候选方案的安全协作者。
4. 本地优先是底线：secret 明文、内部架构、审计证据默认不出本机。
5. 治理系统必须能长期记忆，但记忆的是结构化安全事实，不是无边界保存敏感明文。

## 总体架构

### 1. 文档层

继续以本地 Markdown 为核心载体，覆盖：

- 架构安全评审
- 接口安全审计
- 依赖与第三方服务审查
- 部署与运维加固记录
- 应急预案
- 事故复盘
- 合规说明
- 安全例外申请
- 密钥轮换记录

每篇文档都应有结构化元数据：

- `document_type`
- `owners`
- `system_tags`
- `security_level`
- `review_status`
- `last_audited_at`
- `risk_summary`
- `linked_assets`
- `linked_findings`

### 2. 风险层

风险从文档中抽取，但不只存在于文档中。风险应当有独立生命周期：

- `open`：已发现，未处理
- `acknowledged`：已确认属实
- `accepted`：风险接受，有原因和到期时间
- `fixing`：处理中
- `fixed`：已修复
- `reviewing`：等待复核
- `closed`：复核关闭
- `reopened`：再次出现或修复失效

风险对象需要记录：

- 文档位置
- 证据摘要
- 遮罩后的 evidence
- 影响资产
- 关联接口、服务、依赖或 secret
- 修复建议
- 负责人
- 截止日期
- 处理历史
- 复核结论

### 3. 资产层

系统需要从文档中逐步沉淀团队安全资产：

- 服务与模块
- API 接口
- 数据库
- 第三方服务
- 域名与 URL
- 关键配置
- secret 占位符
- 部署环境
- 依赖包
- 业务数据类型

资产不要求第一版手动建模完整；可以从文档、风险和 secret 引用中半自动抽取。

### 4. 知识层

长期积累安全知识和规范：

- 内部安全规范
- 审计 checklist
- 修复模板
- 例外审批模板
- 应急响应模板
- 常见风险解释
- 团队历史事故模式

AI 优先使用本地知识层解释和建议；只有用户明确配置外部模型时，才通过脱敏上下文调用模型。

### 5. 治理层

将风险转化为团队可执行流程：

- 风险看板
- 负责人分配
- 到期提醒
- 复核队列
- 版本风险 diff
- 审计报告
- 安全例外台账
- 合规映射

第一阶段可以不做账号体系，先以本地负责人文本字段实现；后续再考虑团队协作同步。

## 核心数据模型

建议新增或扩展 Rust 核心模块：

- `crates/rust_tool_core/src/memo/audit.rs`：风险扫描与 finding 模型
- `crates/rust_tool_core/src/memo/governance.rs`：治理对象、状态流转、负责人和复核
- `crates/rust_tool_core/src/memo/assets.rs`：资产抽取与资产索引
- `crates/rust_tool_core/src/memo/knowledge.rs`：本地规范、模板、checklist
- `crates/rust_tool_core/src/memo/history.rs`：文档版本、风险变更和审计事件

### SecurityCase

`SecurityCase` 是长期治理系统的核心对象，用于承载一个风险、例外、复核或安全任务。

```rust
pub struct SecurityCase {
    pub id: String,
    pub case_type: SecurityCaseType,
    pub title: String,
    pub severity: FindingSeverity,
    pub status: SecurityCaseStatus,
    pub source_doc_id: String,
    pub source_finding_id: Option<String>,
    pub linked_assets: Vec<String>,
    pub owner: Option<String>,
    pub due_at: Option<String>,
    pub accepted_until: Option<String>,
    pub rationale: Option<String>,
    pub created_at: String,
    pub updated_at: String,
    pub events: Vec<SecurityCaseEvent>,
}
```

### SecurityAsset

```rust
pub struct SecurityAsset {
    pub id: String,
    pub asset_type: SecurityAssetType,
    pub name: String,
    pub aliases: Vec<String>,
    pub tags: Vec<String>,
    pub source_doc_ids: Vec<String>,
    pub linked_secret_keys: Vec<String>,
    pub linked_case_ids: Vec<String>,
    pub last_seen_at: String,
}
```

### AuditEvent

所有关键动作都进入本地审计事件流：

```rust
pub struct AuditEvent {
    pub id: String,
    pub event_type: AuditEventType,
    pub actor: String,
    pub target_id: String,
    pub summary: String,
    pub created_at: String,
    pub metadata: serde_json::Value,
}
```

事件必须避免保存 secret 明文。

## 功能路线图

### Phase 6：安全档案首页

目标：让用户进入模块后看到团队安全态势，而不仅是文档列表。

交付内容：

1. 新增安全档案首页视图。
2. 显示风险概览：
   - 高危 / 警告 / 信息数量
   - 待复核数量
   - 即将到期风险接受
   - 最近新增风险
3. 显示资产概览：
   - 服务数量
   - 接口数量
   - secret 占位符数量
   - 第三方服务数量
4. 显示最近活动：
   - 文档更新
   - 风险状态变化
   - 复核记录
   - 导出报告

验收标准：

- 首页数据全部来自本地资料库。
- 无文档时显示空状态和创建入口。
- 首页点击风险或资产可跳转到对应文档或详情。

### Phase 7：风险生命周期与复核流

目标：让风险从“被发现”变成“可治理的任务”。

交付内容：

1. 新增 `SecurityCase` 持久化。
2. 风险卡片支持创建治理任务。
3. 支持状态流转：
   - 确认风险
   - 分配负责人
   - 标记修复中
   - 提交复核
   - 关闭
   - 重开
4. 支持风险接受：
   - 接受原因
   - 到期时间
   - 到期后自动回到待处理队列
5. 每次状态变化写入 `AuditEvent`。

验收标准：

- 风险状态变化可追溯。
- 风险接受必须填写原因和到期时间。
- 已关闭风险在同一位置再次出现时可以被自动重开。

### Phase 8：资产索引与关系图

目标：把散落在文档中的服务、接口、secret、第三方依赖连接起来。

交付内容：

1. 从 Markdown 中抽取资产：
   - URL
   - API 路径
   - 数据库连接描述
   - secret 占位符
   - 服务名与模块名
2. 建立资产详情页：
   - 出现在哪些文档
   - 关联哪些风险
   - 关联哪些 secret 占位符
   - 最近一次审计时间
3. 建立轻量关系图：
   - 文档 -> 资产
   - 资产 -> 风险
   - 资产 -> secret
   - 风险 -> 修复记录

验收标准：

- 搜索一个服务或 secret key，可以看到相关文档和风险。
- 删除或重命名文档后，资产索引能更新。
- 关系图不展示 secret 明文。

### Phase 9：版本风险 diff

目标：让团队知道一次文档修改新增了什么风险、修复了什么风险。

交付内容：

1. 保存文档时记录轻量版本快照或内容 hash。
2. 对比上次扫描结果和当前扫描结果。
3. 展示：
   - 新增风险
   - 消失风险
   - 等级变化
   - 位置变化
4. 在保存确认或审计面板中提示风险变化。

验收标准：

- 修改文档后能看到风险 diff。
- 修复后消失的风险不会直接丢失历史。
- 报告中可以包含风险变化摘要。

### Phase 10：规范映射与安全 checklist

目标：把风险和文档连接到团队规范，形成持续审计标准。

交付内容：

1. 新增本地规范库：
   - 内部安全规范
   - OWASP 类别
   - 接口安全 checklist
   - 密钥管理 checklist
   - 部署安全 checklist
2. 风险规则关联规范条目。
3. 文档类型自动推荐 checklist。
4. AI 可根据当前文档生成缺失项提醒。
5. 审计报告显示规范映射。

验收标准：

- 每个核心风险类型能映射至少一个规范条目。
- 用户可以查看“为什么这是风险”及对应规范依据。
- checklist 完成状态可保存。

### Phase 11：安全例外与到期治理

目标：现实中不是所有风险都能立即修复，系统必须能管理例外，而不是让例外消失。

交付内容：

1. 支持从风险创建安全例外。
2. 例外必须包含：
   - 接受原因
   - 影响范围
   - 补偿控制
   - 到期时间
   - 复核人
3. 首页显示即将到期和已过期例外。
4. 到期后自动提醒重新评估。

验收标准：

- 不能创建无到期时间的安全例外。
- 例外导出报告中必须单独列出。
- 例外不会让原始风险历史消失。

### Phase 12：审计报告中心

目标：让系统能输出面向团队、评审或归档的安全报告。

交付内容：

1. 支持按范围生成报告：
   - 单篇文档
   - 单个资产
   - 一组标签
   - 最近时间范围
2. 报告内容：
   - 风险摘要
   - 关闭与未关闭风险
   - 风险接受台账
   - 规范映射
   - 版本风险变化
   - 复核记录
3. 默认导出 Markdown。
4. 后续支持 PDF 或 HTML。

验收标准：

- 导出报告不包含 secret 明文。
- 报告能被重新导入或与本地 case 关联。
- 报告生成记录写入 `AuditEvent`。

## 存储策略

推荐保持本地资料库可读、可备份、可迁移：

```text
rust-tool/
  documents/
  governance/
    cases.jsonl
    events.jsonl
    accepted-risks.json
  indexes/
    assets.json
    findings.json
    doc-history.json
  standards/
    builtin/
    custom/
  reports/
  secrets.kdbx
```

原则：

1. 文档仍是 Markdown 文件。
2. 治理事件使用 JSONL，便于追加和恢复。
3. 索引可以重建，不作为唯一事实来源。
4. secret 明文只在 KDBX 中，不进入 JSON、Markdown 报告或索引。
5. 备份恢复必须包含治理数据和索引，但恢复后可以重建索引。

## AI 能力边界

AI 在长期治理系统中的角色：

- 归纳文档类型
- 提取资产候选
- 解释风险
- 生成修复建议
- 生成 checklist
- 对比版本变化
- 草拟审计报告
- 总结复核意见

AI 不应自动完成：

- 关闭风险
- 接受风险
- 删除审计历史
- 导出包含 secret 明文的报告
- 绕过复核直接批量修改文档

默认策略：

1. 本地规则扫描优先。
2. AI 输入使用脱敏上下文。
3. 外部模型调用显示明显状态。
4. 高风险 AI 权限继续沿用现有 `allowAiSecrets` 设置。

## 开发优先级

推荐顺序：

1. 完成 `07` 的 Phase 1 + Phase 2，形成可靠文档工作台和本地风险扫描。
2. 开始 `08` 的 Phase 6，增加安全档案首页。
3. 实施 Phase 7，把风险升级为可追踪 case。
4. 实施 Phase 8，建立资产索引。
5. 实施 Phase 9，增加版本风险 diff。
6. 实施 Phase 10 到 Phase 12，完善规范、例外和报告。

最小可用治理版本应包含：

- 安全档案首页
- 风险 case 生命周期
- 本地审计事件
- 资产索引
- 单篇文档审计报告

## 验收标准

达到“持续陪团队做安全治理的 AI 安全档案系统”的最低标准：

1. 团队能知道当前有哪些未关闭安全风险。
2. 每个风险能追溯来源、证据、处理历史和复核结论。
3. 文档改动能显示新增和修复的风险。
4. 系统能回答“某个 secret / 服务 / 接口出现在哪些文档和风险中”。
5. 安全例外有原因、有到期、有复核记录。
6. 审计报告可以脱敏导出并归档。
7. secret 明文默认不会进入 AI、索引、报告或日志。

## 长期愿景

当系统成熟后，它应该能回答这些问题：

- 我们当前最高风险的 5 个问题是什么？
- 哪些风险已经逾期？
- 哪些服务使用了第三方 HTTP 链接？
- 哪些文档提到了同一个 secret 占位符？
- 本周新增了哪些风险？
- 某次架构变更修复了哪些风险，又引入了哪些风险？
- 哪些风险被接受了，接受理由和到期时间是什么？
- 这篇文档如果要提交安全评审，还缺哪些 checklist？

这才是目标产品的核心价值：不是替团队“写安全文档”，而是让团队的安全知识、风险和决策长期有地方沉淀、被看见、被追踪、被复核。

## 2026-06-11 实施复盘与目标判断

08 的核心方向已经从路线图进入第一版系统能力：

1. 安全档案首页已落地，进入模块后能看到未关闭风险、待复核、资产数量和最近活动。
2. 风险已升级为 `SecurityCase`，支持确认、处理中、复核、关闭、重开和接受。
3. 风险接受必须记录原因、影响范围、补偿控制、复核人和到期时间；到期后会重新进入治理队列。
4. 本地审计事件已记录关键动作，风险状态、报告生成、安全分享等行为可以追溯。
5. 资产索引已覆盖服务、接口、URL、Secret、数据库、依赖、部署环境和业务数据类型，并能生成轻量关系图。
6. 文档保存已建立风险 diff 基线，能识别新增、修复、移动和等级变化。
7. 本地规范和 checklist 已接入风险、资产和文档上下文，能形成推荐复核项。
8. 报告中心已支持按全部、文档、资产、标签和时间范围生成脱敏 Markdown 报告。

因此，对“07，08 实施后就是我们心中完美的团队安全治理档案系统了吗？”的判断是：

不是最终完美版，但已经是“可持续治理系统的第一版闭环”。它不再只是文档编辑器，也不只是风险扫描器，而是已经具备档案首页、风险生命周期、资产地图、规范复核、版本风险 diff、报告导出和安全分享的一体化本地系统。

后续要逼近“心中完美”，重点不是继续堆按钮，而是补齐长期团队治理的深水区：

1. 多人协作语义：负责人、复核人、外部 issue 或 PR 的映射。
2. 趋势和节奏：按周、按系统、按资产统计风险新增、关闭、逾期和复发。
3. 知识层沉淀：把团队规范、历史事故、修复模板和例外模板做成可维护的本地知识库。
4. 更强关系图：让文档、资产、风险、Secret、报告、复核事件之间可以按问题反查。
5. 更可解释的 AI：所有 AI 建议都应能说明依据、引用本地规范，并保持用户确认后才执行。
6. 更完整的报告生命周期：报告生成、归档、对比、复用和重新导入。

下一步建议以 `09` 计划承接：不要把目标命名为“更多功能”，而是命名为“团队安全治理运营层”。这一层要回答：谁负责、何时到期、趋势是否变好、哪些系统反复出问题、哪些例外需要重新评估。
