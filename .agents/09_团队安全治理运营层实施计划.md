# 团队安全治理运营层实施计划

## 目标定位

07 和 08 已经把 AI 安全文档模块推进到“本地安全档案闭环”：能扫描风险、沉淀 case、抽取资产、映射 checklist、生成报告。

09 的目标不是继续增加零散功能，而是让系统开始承担团队安全治理的日常运营问题：

1. 谁负责这个风险？
2. 什么时候到期？
3. 这个系统的风险趋势是否变好？
4. 哪些风险反复出现？
5. 哪些安全例外需要重新评估？
6. 哪些规范、事故和修复模板应该沉淀成团队知识？

## 产品原则

1. 运营层优先回答“下一步该做什么”，而不是只展示“发生了什么”。
2. 所有趋势和提醒都来自本地事件、case、资产和报告，不依赖外部服务。
3. AI 可以帮助归纳、解释和草拟，但不能自动关闭风险、接受例外或删除历史。
4. 团队协作先用轻量字段和链接表达，不急着引入账号体系。
5. 继续保持 secret 明文不进入索引、报告、AI 上下文和事件日志。

## Phase 13：治理任务台

目标：把风险 case 变成团队每天能处理的任务列表。

交付内容：

1. 新增任务视图：
   - 我的待办
   - 即将到期
   - 逾期风险
   - 待复核
   - 已接受但即将到期的例外
2. 支持按 owner、reviewer、status、severity、asset、tag 过滤。
3. case 增加外部链接字段：
   - issue 链接
   - PR 链接
   - 评审会议链接
   - 证据附件路径或报告路径
4. 首页增加“下一步建议”列表，优先排序：
   - 逾期高危
   - 即将到期例外
   - 重开风险
   - 无 owner 的高危风险

验收标准：

- 用户能在 30 秒内看到当前最该处理的 5 件事。
- 每个待办都能跳回原文档、资产、case 和事件历史。
- 无 owner 或无到期时间的高危风险会被显著提示。

## Phase 14：趋势与复发分析

目标：让团队知道治理是否真的在变好。

交付内容：

1. 增加趋势统计：
   - 最近 7 / 30 / 90 天新增风险
   - 关闭风险
   - 重开风险
   - 接受风险
   - 逾期风险
2. 增加维度拆分：
   - 按资产
   - 按文档类型
   - 按风险类型
   - 按 owner
   - 按标签
3. 增加复发分析：
   - 同一 finding fingerprint 再次出现
   - 同一资产反复出现同类风险
   - 已关闭风险在相近位置重开
4. 报告中增加趋势摘要和复发风险清单。

验收标准：

- 系统能回答“本周新增了哪些高危风险”。
- 系统能回答“哪个服务最近反复出现密钥治理问题”。
- 趋势数据不依赖不可恢复的缓存，能从事件和 case 重建。

## Phase 15：团队安全知识库

目标：把团队规范、历史事故、修复模板和例外模板沉淀为 AI 可引用的本地知识。

交付内容：

1. 新增知识条目模型：
   - 规范条款
   - 修复模板
   - 例外模板
   - 事故模式
   - 审计提问模板
2. 知识条目支持标签：
   - risk kind
   - asset type
   - document type
   - severity
   - team
3. AI 解释风险时优先引用本地知识条目。
4. checklist 和报告可以引用知识条目来源。
5. 支持从已关闭 case 生成候选知识条目，但必须用户确认后保存。

验收标准：

- 风险解释能显示“依据来自哪条本地规范或模板”。
- 用户可以把一次有效修复沉淀为模板。
- AI 不会把 secret 明文写入知识条目。

## Phase 16：报告生命周期

目标：报告不是一次性导出文件，而是可归档、可对比、可追溯的治理证据。

交付内容：

1. 报告生成后写入报告索引：
   - report id
   - scope
   - filters
   - generated_at
   - linked case ids
   - linked asset ids
   - content hash
2. 支持报告详情页：
   - 查看 Markdown
   - 关联 case
   - 关联资产
   - 重新生成同范围报告
3. 支持两份报告对比：
   - 风险新增
   - 风险关闭
   - 接受风险变化
   - checklist 变化
4. 报告导出继续默认脱敏。

验收标准：

- 用户能找到某次安全评审对应的报告和当时的 case 状态。
- 再次生成同范围报告时能看到变化摘要。
- 报告索引不保存 secret 明文。

## Phase 17：轻量团队协作适配

目标：在不引入完整账号系统的前提下，让本地档案能接入团队实际工作流。

交付内容：

1. owner 和 reviewer 支持本地联系人别名。
2. 支持配置外部系统链接模板：
   - GitHub issue
   - GitHub PR
   - Jira
   - 飞书 / Slack 讨论链接
3. case 支持复制安全摘要：
   - 不含 secret 明文
   - 包含风险标题、等级、证据摘要、建议、链接
4. 可选生成外部 issue 草稿，但默认不自动发送。

验收标准：

- 用户可以把一个 case 转成可复制的团队任务说明。
- 外部链接只作为引用，不成为本地事实来源。
- 任何向外部系统发送数据的动作都必须由用户确认。

## 开发顺序

1. Phase 13：先做治理任务台，因为它直接提升日常可用性。
2. Phase 14：再做趋势和复发分析，让团队看到治理效果。
3. Phase 15：补知识库，让 AI 建议从“聪明”变成“有依据”。
4. Phase 16：完善报告生命周期，满足复盘和归档。
5. Phase 17：最后接轻量团队协作，避免过早引入外部依赖。

## 最小验收闭环

09 完成后，系统应能稳定回答：

1. 今天最该处理的风险是哪几个？
2. 哪些风险快到期或已经逾期？
3. 哪些资产最近风险在上升？
4. 哪些问题修过又复发了？
5. 这次报告相比上次有什么变化？
6. 当前 AI 建议依据了哪条团队规范或历史模板？

这一步之后，RustTool 的 AI 安全文档模块才会更接近我们心中那个“持续陪团队做安全治理的 AI 安全档案系统”：它不仅帮团队写和审，还能陪团队长期运营安全治理。
